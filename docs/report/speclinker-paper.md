## A-2. 비판의 종합

세 줄로 요약하면:
1. **측정의 부재** — 충실도·AIDD 결과·SOP 가독성 모두 정량 근거가 없다(결함 1·2·6). 이것이 가장 큰 약점이다.
2. **JIT의 입구와 출구가 약하다** — 엔티티 추출(입구)과 ripple 랭킹(출구)이 가장 덜 엔지니어링되어 있다(결함 3·4).
3. **이중 역할은 매력적이나 경험적으로 미검증** — 특히 인간 SOP 측면(결함 6·10).

> **그럼에도** 측정 가능한 형태의 가치(그래프 탐색성·소스앵커 무손실 그라운딩)는 이미 동작하며(JT_CODE→176 ripple), 이는 Part B의 효과성 논증의 *검증된* 토대가 된다.

# Part B. 효과성 논증 (논문 형식)

## 초록 (Abstract)

운영 중 시스템(SM, System Management)의 유지보수·변경 개발에서 대규모 언어모델(LLM) 기반 개발(AIDD)을 적용할 때, 핵심 난점은 **LLM이 시스템의 현행(AS-IS)을 알지 못한다**는 점이다. 소스코드는 존재하나, 비즈니스 규칙은 여러 레이어(컨트롤러·서비스·DAO·SQL·배치)에 분산되어 *단일 파일 단위로는 판단이 불가능*하다. 본 연구는 운영 소스를 역추출하여 **하나의 스펙 산출물이 (i) 인간이 이해하는 SOP 수준의 참조 문서이자 (ii) LLM에 주입 가능한 JIT(Just-In-Time) 그라운딩 콘텐츠로 동시에 기능**하도록 설계한 플러그인 *Speclinker*를 제시한다. 핵심 설계는 ① 사실(facts)은 결정론적 스크립트로 zero-token 생성하고 의미(semantics)만 LLM이 보강하는 하이브리드, ② 산출물을 "인간용 서술 + 기계용 ID·교차링크·소스앵커"의 이중 레이어로 구성, ③ 변경 시 요약 주입 대신 **그래프 기반 영향슬라이스 + 소스앵커의 실코드 JIT 판독**으로 그라운딩하는 것이다. 운영 어드민 1종에 대한 적용에서 657개 API 명세와 270개 테이블 명세가 자동 역생성되었고, 공통코드 테이블 변경 시 영향받는 176개 API가 자동 식별되었다. 본 논문은 이 이중 역할이 AIDD를 실현 가능하게 한다는 명제를 논증하고, 그 타당성 위협과 검증 설계를 제시한다.

## 1. 서론 (Introduction)

### 1.1 문제

SM 사업의 변경 개발은 다음 조건에서 수행된다: (a) 시스템이 이미 운영 중이고, (b) 설계 문서는 부재하거나 현행과 어긋나며, (c) 변경 요구는 외부(Jira SR, 고객 PPT/Word)에서 들어오고, (d) 잘못된 변경은 운영 장애로 직결된다. 이때 LLM에게 "이 SR을 구현하라"고 지시하면, LLM은 변경 대상 영역의 *현행이 어떻게 동작하는지*를 모른 채 코드를 생성한다. 이는 단순 신규개발(green-field)과 근본적으로 다른 난점이다.

### 1.2 기존 접근의 한계

(1) **소스 직접 판독**: 비즈니스 규칙이 레이어를 가로질러 분산되어, 단일 파일·단일 함수 판독으로는 현행 규칙·영향 범위를 재구성할 수 없다. (2) **사람이 작성한 설계 문서**: 작성 비용이 크고 즉시 stale 되며, 기계가 소비하도록 구조화되어 있지 않다. (3) **요약 스펙의 LLM 주입**: 요약은 소스의 손실 압축이므로, 변경의 근거로 삼기엔 디테일·엣지케이스가 증발하고(lossy) 시점이 고정된다(stale).

### 1.3 기여 (Contributions)

- **C1.** 운영 소스의 역추출 산출물을 *인간 SOP*와 *LLM JIT 콘텐츠*로 이중 활용하는 설계 원칙을 제시한다(§4).
- **C2.** 사실은 결정론적 생성(zero-token), 의미만 LLM 보강하는 하이브리드 생성으로 비용을 절감한다(§3.2).
- **C3.** 요약 주입을 대체하는 **그래프 영향슬라이스 + 소스앵커 JIT 판독** 그라운딩을 제시하고, 공유 자원 변경의 ripple을 자동 식별함을 보인다(§3.3, §5).
- **C4.** 위 주장의 타당성 위협과 이를 검증하기 위한 평가 설계를 명시한다(§5.2, §6).

## 2. 배경 및 관련 연구 (Background)

- **SDD (Spec-Driven Development)**: 스펙을 1차 산물로 두고 개발을 구동. 본 연구는 SDD의 "스펙"을 *역추출된 현행*으로 확장한다.
- **BMAD-METHOD**: 에이전트 기반 애자일. story = 자기완결 컨텍스트 단위, 품질 게이트(PASS/CONCERNS/FAIL). 본 연구의 AIDD 루프가 이를 차용한다.
- **RAG / Code Grounding**: 검색-증강 생성. 본 연구의 JIT 그라운딩은 *구조화 그래프 검색 + 소스앵커 정밀 판독*이라는 점에서, 비정형 임베딩 검색과 구별되는 결정론적 그라운딩이다.
- **Reverse Engineering / Program Comprehension**: 정적 분석으로 호출체인·데이터 흐름 복원. 본 연구는 정적 분석을 *LLM 주입 가능한 산출물*로 합성한다는 점에서 차별화된다.

## 3. 시스템 설계 (Method)

### 3.1 파이프라인 개요

```
운영 소스 ──RECON──▶ 스펙 코퍼스(INF/SCH/UIS/FUNC) ──┬──▶ 인간: SpecLens(Docsify) 열람 / 납품
                                                      └──▶ 기계: 그래프 + 소스앵커 ──▶ AIDD/변경 JIT 주입
```

- **RECON(역추출)**: tree-sitter 기반 제로-LLM 정적 스캔으로 라우터·서비스·DAO·SQL 구조 추출 → 호출체인 합성으로 분산된 규칙 통합 → Claude 에이전트가 의미 명세화. 대량 생성은 독립 서브프로세스 디스패처로 병렬화하여 메인 컨텍스트 오염을 방지하고 실패를 멱등 재시도한다.
- **산출물**: INF(API), SCH(DB), UIS(화면), FUNC(추적 축). 각 산출물은 `frontmatter(ID·링크)` + `본문(서술)` + `근거 소스 앵커(file:line)`를 가진다.

### 3.2 하이브리드 생성 — 사실/의미 분리

핵심 비용 원리: **사실은 LLM이 필요 없다.** 컬럼·타입·인덱스·FK·ERD·교차링크는 파일 파싱(+선택적 DB 드라이버)으로 결정론적 생성하고, *의미*(코드값의 뜻, 비즈니스 주의사항, 한글 설명)만 LLM이 마커를 채운다. SCH 명세에서 이 분리로 사실 부분(약 70%, 추정)을 토큰 0으로 생성한다. 이는 정확도(사실은 권위 소스에서)와 비용(의미만 과금)을 동시에 만족한다.

### 3.3 JIT 그라운딩 — 요약 주입의 대체

변경 시 그라운딩 절차:
1. SR + 첨부(PPT/Word/Excel 추출)에서 엔티티(테이블·API·경로 키워드) 도출.
2. 스펙 frontmatter에서 그래프 구축 — forward(INF→table)·**reverse(table→INF)**. 사전 계산 그래프가 없어도 즉석 빌드.
3. 엔티티로 영향슬라이스 특정 + 공유 자원 사용처(ripple) 확장.
4. 영향 항목의 **소스앵커(file:line)를 LLM이 직접 판독** → 최신·무손실 현행 확보.

이때 LLM에 주입되는 것은 *요약 본문이 아니라* **영향 인덱스 + 소스 위치**이며, 실제 의미는 LLM이 그 자리에서 실코드를 읽어 획득한다. 따라서 lossy·stale 문제가 구조적으로 회피된다(스펙은 "어디를 볼지"의 인덱스, 소스는 "무엇인지"의 진실).

## 4. 핵심 명제 — 이중 역할 산출물 (The Dual-Role Thesis)

> **명제.** 운영 소스의 역추출 산출물은, "인간용 서술 레이어"를 "기계용 ID·교차링크·소스앵커 레이어" 위에 적층하고, **LLM 소비 경로가 산문 요약이 아니라 구조 레이어(그래프)로 검색하고 소스앵커로 그라운딩**하도록 설계될 때, 단일 산출물로 (i) 신규입사자·신규요건 분석을 위한 SOP 수준 참조와 (ii) AIDD를 위한 JIT 그라운딩을 동시에 충족한다.

이 명제가 성립하면, AIDD는 다음 의미에서 *실현 가능*해진다:
- **신규입사자/신규요건 분석**: 사람은 SpecLens에서 도메인·API·스키마·화면을 탐색하며 현행을 이해한다(서술 레이어).
- **AIDD 변경**: 동일 산출물의 구조 레이어가 변경 영향을 정밀 특정하고, 소스앵커가 LLM에 정확한 현행을 공급한다 → LLM이 "현행을 알고" 변경한다.

**왜 단일 산출물이어야 하는가.** 인간용·기계용을 분리 생성하면 두 배의 비용과 두 배의 stale 위험이 생긴다. 동일 frontmatter/앵커가 양쪽을 구동하면, 한 번의 RECON으로 납품물과 AI 지식을 동시에 얻고 동기화 비용이 0이 된다.

## 5. 평가 (Evaluation)

> 본 절은 **현재 측정된 결과**와 **명제 검증을 위해 제안하는 평가 설계**를 *명확히 구분*한다. 후자는 미수행이며, Part A의 결함 1·2·6에 대응한다.

### 5.1 현재 측정된 결과 (Measured)

| 지표 | 값 | 의미 |
|------|----|------|

이 결과들은 **기계 소비 가능성(machine-traversability)과 무손실 그라운딩의 토대**가 동작함을 보인다. 특히 ripple 자동 식별(176)은 명제의 (ii) 측면에 대한 직접 증거다.

### 5.2 제안하는 평가 설계 (Proposed — 미수행)

명제를 입증하려면 다음 가설을 검정해야 한다:

- **H1 (충실도)**: 전문가 주석 정답셋 대비, 추출된 비즈니스 규칙·코드값·테이블 사용처의 precision/recall ≥ 임계치. *(결함 1)*
- **H2 (SOP 가독성)**: 신규입사자가 스펙만으로 시스템 질문(데이터 흐름·영향 범위)에 답하는 정확도·소요시간이, 스펙 없이(소스만) 대비 유의하게 개선. *(결함 6)*
- **H3 (JIT 효능)**: 변경 영향 브리프의 *재현율*(실제 영향 컴포넌트를 누락 없이 포함) 및 ripple *정밀도*(상위 K개의 실제 위험 적중률). *(결함 3·4)*
- **H4 (AIDD 결과)**: 동일 FUNC에 대해 (풀 그라운딩) vs (소스만) 구현 시 테스트 통과율·결함률 비교. *(결함 2)*
- **H5 (비용)**: 스펙당·변경당 토큰/시간, 엔터프라이즈 규모 ROI. *(결함 9)*

> **정직한 위치 표명.** 현재 *측정된* 것은 명제의 기계 측면(검색·앵커·ripple)이다. *인간 SOP 측면과 end-to-end AIDD 결과*는 **논증되었으나 미검증**이며, H1·H2·H4가 이를 입증하기 전까지 효과성 주장은 *조건부*다.

## 6. 논의 및 타당성 위협 (Discussion & Threats to Validity)

- **구성 타당도**: "657 생성"은 산출 *양*이지 *질*이 아니다(결함 1). 질은 H1으로만 입증된다.
- **내적 타당도**: qa-agent가 스펙을 오라클로 쓰면 순환 검증이 된다(결함 2). 독립 오라클(테스트·전문가 리뷰)이 필요하다.
- **외적 타당도**: 단일 Java/MyBatis 프로젝트 검증은 일반화를 보장하지 않는다(결함 8). 다중 스택 재현이 요구된다.
- **신호 품질**: ripple 176은 재현율 증거이자 *정밀도 부족*의 증거다 — 편재 자원에서 신호가 노이즈화된다(결함 4). 랭킹이 없으면 실행 불가능하다.
- **현행성**: 그래프가 stale 스펙에서 빌드되면 영향슬라이스가 틀린다(결함 5). JIT 소스 판독은 *읽은 항목*의 현행성만 보정하며, *누락된 항목*은 보정하지 못한다.
- **천장(ceiling)**: 의도/why, 런타임 암묵 의존(동적 SQL·이벤트)은 정적 역추출의 사각지대다. 본 방법은 이를 *제거*하지 못하고 소스앵커 판독으로 *부분 완화*할 뿐이다.

## 7. 결론 및 향후 연구 (Conclusion)

본 연구는 운영 시스템의 역추출 산출물을 인간 SOP이자 LLM JIT 콘텐츠로 이중 활용하여 AIDD를 실현하려는 설계를 제시했다. **검증된 것**은 사실/의미 분리에 의한 비용 절감, 스펙으로부터의 그래프 구성 가능성, 공유 자원 ripple의 자동 식별이다. **미검증이나 논증된 것**은 인간 SOP 가독성과 end-to-end AIDD 결과 품질이다. 효과성의 최종 입증은 §5.2의 H1–H5, 특히 충실도(H1)·AIDD 결과(H4)·SOP 가독성(H2)의 측정에 달려 있다.

향후 연구: ① 엔티티 자동 추출의 구조화와 재현율 측정, ② ripple 관련도 랭킹 및 전이 확장, ③ drift 게이트화로 현행성 강제, ④ 충실도·AIDD 결과의 통제 실험, ⑤ 의도/암묵의존 보강을 위한 동적 추적의 통합.
