# Part B. 효과성 논증 (논문 형식)

## 초록 (Abstract)

운영 중 시스템(SM)의 변경 개발에 LLM 기반 개발(AIDD)을 적용할 때 핵심 난점은 **LLM이 현행(AS-IS)을 모른다**는 점이다. 소스는 있으나 비즈니스 규칙은 컨트롤러·서비스·DAO·SQL·배치에 분산되어 단일 파일로는 판단 불가하고, 쿼리의 코드값(`'03'`)처럼 *소스에 의미가 없는* 신호도 많다. 본 연구는 **소스를 진실로 두고, 역추출 산출물을 소비자별 레이어로 분리**한다: (i) 기계용 = 얇은 구조 인덱스 + **전체 호출체인 앵커** + DB로 해소한 코드 의미 → 변경 시 그래프로 영향슬라이스를 특정하고 소스앵커로 실코드를 JIT 판독, (ii) 인간용 = 개념·워크플로우 내러티브(코드 의미 주입) → 온보딩. 두 레이어는 한 번의 RECON이 만든 동일 substrate를 공유한다. 운영 어드민 1종에서 657 API·270 테이블 명세가 자동 역생성됐고, 공통코드 변경 영향이 자동 식별·**관련도 랭킹/격리**되며, product 쿼리의 코드 컬럼 195개 중 58개의 코드 그룹이 소스에서 정적 복원됐다. 본 논문은 brownfield에서 SDD가 *AS-IS 서술 + TO-BE 처방*의 두 시제로 작동함을 정리하고, 이중 레이어가 AIDD를 실현 가능하게 한다는 명제를 논증한다.

## 1. 서론

### 1.1 문제
SM 변경 개발은 (a) 시스템이 운영 중, (b) 설계문서 부재/stale, (c) 변경요구가 외부(Jira SR·고객 PPT/Word), (d) 오변경=운영장애의 조건에서 수행된다. LLM에 "이 SR 구현하라" 하면 *현행 동작*을 모른 채 코드를 생성한다. 더해, 쿼리의 `WHERE STS_CD='03'`처럼 **소스에 코드의 의미가 없어** 의도 파악이 불가한 영역이 존재한다.

### 1.2 기존 접근의 한계
(1) 소스 직접 판독 — 규칙이 레이어 분산, 코드값 의미 부재. (2) 사람 작성 문서 — 비용·stale·비구조화. (3) 요약 스펙 LLM 주입 — 손실 압축(lossy)·시점 고정(stale).

### 1.3 기여
- **C1.** brownfield에서 "소스=진실, 스펙=소비자별 레이어"라는 재설계 원칙(§3.4).
- **C2.** 사실=결정론적 zero-token, 의미만 LLM 하이브리드(§3.2).
- **C3.** 그래프 영향슬라이스 + 전체체인 소스앵커 JIT 판독 + **관련도 랭킹/편재 격리**(§3.3, §5).
- **C4.** 쿼리 코드값을 DB/소스 권위로 *사실* 해소해 의도 복원(§3.5).
- **C5.** brownfield SDD의 두 시제(AS-IS 서술/TO-BE 처방) 정리(§4.1).

## 2. 배경 및 관련 연구
- **SDD**: 스펙을 1차 산물로 개발 구동. 본 연구는 SDD가 brownfield에서 *변경(TO-BE)*에 적용되고, *현행(AS-IS)*은 그 그라운딩임을 구분한다(§4.1).
- **BMAD-METHOD**: story 단위 + QA 게이트. AIDD 루프가 차용.
- **RAG/Code Grounding**: 본 JIT는 *구조 그래프 검색 + 소스앵커 결정론 판독*으로 임베딩 검색과 구별.
- **Program Comprehension / Static Call-Graph**: 정적분석을 *LLM 주입 가능 산출물*로 합성.

## 3. 시스템 설계

### 3.1 파이프라인
```
운영 소스 ──RECON──▶ substrate(구조: id·method·path·tables·앵커·코드의미) ──┬──▶ 인간: 내러티브(SOP)
                                                                          └──▶ 기계: 인덱스+앵커 ──▶ JIT 변경
```

### 3.2 하이브리드 생성 (사실/의미)
사실(컬럼·타입·FK·ERD·링크)은 파일파싱(+선택 DB 드라이버)으로 zero-token, 의미만 LLM. SCH에서 사실 ~70%를 토큰 0으로(추정).

### 3.3 JIT 그라운딩 (요약 주입 대체)
SR+첨부 → **엔티티 자동추출** → 스펙 frontmatter서 그래프(forward/reverse) → 영향슬라이스 + **관련도 랭킹 + 편재 공통자원 격리** + ripple → **소스앵커(file:line) 직접 판독**. 주입되는 것은 요약이 아니라 인덱스+소스 위치이며, 의미는 실코드에서 획득 → lossy·stale 구조적 회피.

### 3.4 INF 레이어 재설계 (핵심, v2 신규)
**원칙: 소스=진실. 스펙은 별도 "완전 정본"을 만들지 않는다.**
- **기계 레이어**: frontmatter + **full-chain 앵커**(controller→service→DAO/mapper) + 해소된 코드의미. 본문 비즈규칙은 *짧은 abstract*(완전성 주장 제거). 현행성 = 드리프트 0(실소스 판독).
- **인간 레이어**: 도메인/기능 내러티브(코드의미 주입). 추상화·생략 허용, 개념수준 현행성(약간 stale 관대).
- **현행성 기준을 소비자에 맞춘다**: AI=소스(오변경 방지), 사람=개념(이해 충분).
- 이로써 v1 결함 10(이중역할 긴장)이 *설계 원칙*으로 해소된다.

### 3.5 코드값 → 쿼리 의도 복원 (v2 신규)
쿼리의 코드 리터럴(`COL_CD='20'`)은 소스에 의미가 없다. `scan_code_literals`가 ① 리터럴과 ② **소스의 `JT_CODE WHERE CODE_GRP_ID='X' AND CODE=t.COL` 패턴에서 컬럼→그룹 매핑을 정적 복원**한다. 그룹을 알면 DB 드라이버/MCP로 `JT_CODE` 전체 값·명을 *사실*로 채우고, 모르면 probe-match, 없으면 소스 enum, 그래도 없으면 `[미확인]`. 출처(`JT_CODE.그룹`)를 표기한다. 복원된 의미는 SCH 코드값·**쿼리 의도**·온보딩 설명에 동시 주입된다 → A·B·C를 한 메커니즘으로.

## 4. 핵심 명제

### 4.1 brownfield SDD의 두 시제 (v2 신규)
SDD의 "스펙=진실"은 *방향*에 따라 다르다.
- **신규구축(forward)**: 스펙 먼저 → 코드 파생. 스펙=진실(SDD 정석).
- **brownfield(우리)**: 코드가 이미 운영 중 → **코드=진실**. 역추출 스펙은 *서술(descriptive)*.

따라서 SM에서 SDD는 사라지지 않고 **변경(TO-BE)**에 산다:
```
[현행코드=진실]→RECON→[AS-IS 스펙: 서술·그라운딩]→(변경요구+그라운딩)→[TO-BE 스펙: 처방·진실]→AIDD→[새 코드]→(배포)→[새 진실]→…
```
AS-IS는 **서술적**(코드가 진실), TO-BE는 **처방적**(스펙이 진실=SDD). 모순이 아니라 시제가 다르다. **함정은 서술적 AS-IS를 처방적 정본인 척 다루는 것**(lossy·드리프트의 근원). 둘을 분리하면 SDD(전방)와 그라운딩(후방)이 공존한다. AS-IS 그라운딩은 SDD의 경쟁자가 아니라 *brownfield SDD의 전제조건*이다(현행을 모르면 안전한 TO-BE 불가).

### 4.2 이중 역할 명제
> 역추출 산출물은 (소스=진실) 위에서 **기계 레이어(인덱스+체인앵커+코드의미)와 인간 레이어(내러티브)로 분리**되고, **AI는 그래프 검색+소스앵커 판독으로, 사람은 내러티브로** 소비할 때, 단일 RECON으로 (i) 온보딩 SOP와 (ii) AIDD JIT 그라운딩을 동시 충족한다. 단일 substrate 공유로 동기화 비용 0.

### 5.2 제안하는 평가 (미수행, 하네스 구비)
- **H1 충실도(의미적)**: 전문가 주석 정답셋 대비 비즈규칙·코드값 정확도. *(워크시트 구비, 주석 미수행)*
- **H2 SOP 가독성**: 신규자가 내러티브로 시스템 질문에 답하는 정확도·시간(vs 소스만). *(레이어 구비, 실험 미수행)*
- **H3 JIT 효능**: 영향 브리프 재현율 + ripple 상위 K 정밀도. *(랭킹 구현, 정답셋 측정 잔여)*
- **H4 AIDD 결과**: 풀그라운딩 vs 소스만 A/B 통과율·결함. *(하네스 구비, 실행 미수행)*
- **H5 비용**: 스펙당·변경당 토큰/시간.

> **정직한 위치:** *측정된* 것은 기계 측면(앵커 커버리지·그래프·랭킹·코드복원). *의미적 충실도·SOP 가독성·AIDD 결과*는 논증·하네스까지이며 미측정 → 효과성 주장은 그 범위에서 조건부다.

## 6. 논의 및 타당성 위협
- **구성 타당도**: "657 생성"은 양. v2는 *검증 가능한* 앵커 커버리지·코드복원율로 일부 대체하나, *의미적* 충실도는 H1 필요.
- **내적 타당도**: qa-agent 스펙 오라클=순환. 독립 오라클(테스트·리뷰) 필요(H4).
- **외적 타당도**: 실검증 1스택(Java). 2번째 실스택 잔여.
- **신호 품질**: ripple 랭킹·격리로 정밀도 개선했으나 *정밀도 자체*는 정답셋으로만 입증(H3).
- **현행성**: JIT 소스판독은 *읽은* 항목만 현행 보정, *누락* 항목은 freshness 게이트의 그래프 신선도에 의존.
- **천장**: 의도/why, 런타임 암묵의존(동적SQL·이벤트)은 사각지대. 코드값 의미는 풀지만 "왜 그 규칙인지"는 미해결.

## 7. 결론
brownfield AIDD에서 **소스를 진실로 두고 역추출 산출물을 소비자별 레이어로 분리**하는 재설계를 제시했다. SDD는 폐기되지 않고 *TO-BE(처방)*에 살아 있으며, AS-IS(서술)는 그 전제조건이다. **검증된 것**: 사실/의미 분리 절감, 그래프 즉석 구성, ripple 자동식별·랭킹·격리, 코드값 의도 정적복원, 앵커 커버리지 측정. **미검증(하네스 구비)**: 의미적 충실도(H1)·SOP 가독성(H2)·AIDD 결과(H4). 효과성의 최종 입증은 이 셋의 통제 측정에 달려 있다.

향후: ① full-chain 앵커 재RECON 후 커버리지 재측정 ② 코드값 DB probe 자동화 ③ 충실도·AIDD 통제실험 ④ 2번째 실스택 ⑤ 의도/암묵의존(동적추적) 보강.
