# 충실도 측정 결과 (T3-A, 논문 H1) — 정직한 1차 측정

> ⚠️ **정정(2026-06-05):** 아래 P/R/F1은 "스펙 tables vs SQL 정규식"의 **self-consistency 프록시**이지
> 충실도(정답 대비)가 아니다. 정답 없는 자동추출 2개 비교라 수치 자체는 무의미. **`eval_anchor_coverage.py`(4-4)로 대체** —
> 앵커 체인 커버리지 + 메타 정확도(검증 가능·드리프트 없음·정답 불필요). 아래는 "왜 프록시가 부적절한지"의 기록으로만 보존.


> 도구: `scripts/eval_fidelity.py --auto-tables`. 대상: nkshop-bos-admin RECON 산출(INF 657).
> 측정일: 2026-06-05. 표본: 120 INF 무작위(seed 42).

## 자동 테이블 일치도 (스펙 `tables:` vs 앵커 SQL의 실제 테이블)

| 지표 | 값 |
|------|----|
| 평가 가능 INF | **17 / 120** (SQL 앵커 보유분만) |
| precision | **0.696** |
| recall | **0.175** |
| F1 | **0.279** |
| tp / fp / fn | 55 / 24 / 260 |

## 해석 (미화 없음)

1. **하네스는 동작한다** — 충실도가 이제 *측정 가능*하다(결함 #1 해소의 1단계).
2. **수치는 낮다, 그러나 부분은 측정 한계 탓**:
   - **앵커 커버리지 14%** — INF `근거 소스`가 대부분 컨트롤러(.java)라 SQL 기반 테이블 검증이 17/120만 가능. → INF 앵커에 DAO/Mapper(.xml) 경로도 포함해야 검증 가능 범위가 커진다(개선항목).
   - **recall 0.18 (fn=260)** — 앵커 1개 파일의 SQL에는 스펙 `tables:`가 누락한 테이블이 다수. 단, ① 스펙은 전체 체인 합집합인데 앵커는 단일 파일이라 비교축이 비대칭, ② SQL 정규식이 별칭·서브쿼리·CTE를 테이블로 오추출해 fn 과대 가능. → 비교축 정합(앵커=동일 체인) + SQL 파서 정교화 필요.
   - **precision 0.70** — 스펙이 주장한 테이블의 70%가 앵커 SQL에 실제 등장. 30%는 다른 파일 유래이거나 오추출.

## 결론

- 이 측정은 **"657 생성"이 양이지 질이 아님**을 정량으로 보여준다(논문 §6 구성타당도 위협의 실증).
- 다음 개선: ① INF 앵커에 Mapper(.xml) 경로 포함 → 평가 커버리지↑, ② SQL 테이블 파서 정교화(별칭/CTE 제거), ③ 비즈규칙 충실도는 `--worksheet`로 전문가 주석 후 채점(H1 완성).
- 논문 §5.1에 *측정된* 충실도로 본 수치를 (커버리지·비대칭 caveat와 함께) 반영한다.
