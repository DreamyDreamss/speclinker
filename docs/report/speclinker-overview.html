01문제 — SM 변경개발의 현실

현행을 모른다

레거시 시스템은 소스 한 파일만 봐선 "지금 어떻게 도는지"를 알 수 없다. 비즈니스 규칙이 컨트롤러→서비스→DAO→SQL→배치에 흩어져 있다.

변경 영향이 안 보인다

공통 테이블 하나 바꾸면 어디가 깨지는지 추적 불가. 보이지 않는 곳에서 회귀가 터진다.

문서가 없거나 낡았다

설계서는 부재하거나 현행과 어긋남(stale). 신규 인력 온보딩·고객 보고에 매번 비용.

AI를 써도 현행을 모른다

AIDD를 도입해도 AI가 운영 시스템의 현행 맥락을 모르면 위험한 변경을 만든다. 특히 쿼리는 소스만 봐선 정확히 못 쓴다(조인·상시필터·코드값).

02왜 SPEC 문서인가 — 한 substrate, 세 청중

운영 시스템에 설계문서가 없으면 그 공백 비용은 세 부류 모두에게 전가된다. 한 번의 역추출(RECON)로 셋을 동시에 충족 — 이것이 도구를 만든 이유.

① 신규 입사자 온보딩

코드 정글에서 "이 화면→어떤 API→어떤 테이블·규칙"을 스스로 재구성하는 수 주~수개월 비용. FUNC→UIS→INF→SCH 지도를 코드 읽기 전에 손에 쥔다.

② 이해관계자 협업

기획자·PM·고객사는 소스를 못 읽는다. 화면 시나리오(UIS)·기능 명세(SRS)·영향 범위(INF/SCH)가 업무협의·분석·설계의 공통 언어가 된다. 스펙 = 협의·영향분석의 매개체.

③ AIDD의 JIT 기초자료

LLM은 현행(AS-IS)을 모른다. 실데이터·실쿼리에서 뽑은 SCH(타입·조인·코드값·상시필터)가 코드/쿼리 생성 시 마크다운 재파싱 없이 직접 소비하는 구조화 JIT 그라운딩.

핵심: 사람(①②)이 보든 기계(③)가 먹든 사실의 원천은 동일(현행 소스·DB) → 청중별로 따로 만들지 않고 하나의 substrate를 레이어로 렌더. 사람=가독성(접이식), 기계=구조화 JSON.

05활용 오픈소스·기술과 "왜"

BMAD-METHOD

AIDD를 story-driven + QA 게이트로. FUNC=story, 전용 qa-agent가 독립 컨텍스트로 3-Layer(스펙·보안·회귀) 검증. → 자동 폭주 없는 개발 규율.

MCP (Atlassian·DB)

Jira에서 SR 요구사항·첨부 수집, Confluence 게시, DB 스키마 권위 조회(Oracle·DB2·MySQL/MariaDB 내장 MCP). → 실제 업무 흐름·실데이터와 직결.

Docsify → SpecLens

역추출 산출물을 무빌드 웹 뷰어로. 대시보드·INF/SCH/UIS·IA 트리·크로스링크 네비게이션. → 사람용 산출물 열람.

tree-sitter · Playwright · Claude

tree-sitter=제로-LLM 정적 소스 스캔, Playwright=화면 DOM 골격 캡처(보조), Claude(Sonnet/Opus)=소스 판독·의미 합성(권위). → 사실은 무료로, 의미만 LLM.

보조: python-pptx/docx/openpyxl(첨부 추출) · pymysql/psycopg2/oracledb(DB 권위) — 전부 선택 의존성(없으면 우아하게 폴백).

06RECON — 현행을 스펙으로

  • 제로-LLM 정적 스캔 — tree-sitter로 라우터·서비스·DAO·SQL 구조를 토큰 0으로 추출
  • 호출체인 합성 — 컨트롤러→서비스→DAO→쿼리를 따라가 흩어진 비즈니스 규칙을 한 곳에 모음 (소스 한 파일론 안 보이는 것)
  • 디스패처 병렬화 — 독립 서브프로세스로 대량 생성, 메인 컨텍스트 오염 없음, 실패 자동 재시도·멱등
  • 다중 스택 중립 — Java/Spring, Next.js, Python 등 경로 기반 도메인 분류(특정 스택 비종속)

실측(운영 admin 1종): API 657개·테이블 270개 스펙을 자동 역생성.

07산출물 — 이중 용도(사람 + AI)

INF (API 명세)

엔드포인트·요청/응답·비즈니스 규칙·트랜잭션 순서·사이드이펙트 + 근거소스 file:line 앵커

SCH (DB 스키마 = 쿼리 작성 계약)

컬럼·타입·키·인덱스·코드값 + 관찰된 조인(논리 FK)·상시필터(soft-delete·테넌트). MCP=물리사실 + 실쿼리=논리관계 → AIDD가 정확한 JOIN·필터 생성.

UIS (화면 설계 · SOP+JIT)

가이드형 진입(사용자가 메뉴로 실화면) + 에이전트 소스판독 → ①작업 시나리오 SOP ②api_hints·소스앵커로 화면→INF JIT. 캡처는 보조.

FUNC (추적 축)

INF/SCH/UIS를 묶는 기능 단위 — RTM·AIDD의 단일 진실

핵심: 같은 산출물이 사람에겐 읽는 문서, AI에겐 frontmatter·링크·소스앵커·구조화 쿼리패턴(_machine/*.json)으로 직접 소비되는 인덱스.

--왜 라이브 브라우저(CDP/Playwright)인가

INF·SCH는 정적 소스로 충분하지만 화면(UIS)은 런타임 산물 — 권한·결재상태·메뉴 컨텍스트에 따라 위젯이 나타나고 사라진다. 그래서 실제 렌더된 DOM이 필요하다.

스택중립 — DOM은 공통분모

Java JSP·Next.js·Vue 무엇이든 최종 산물은 DOM. 프레임워크별 파서 분기 0 → 렌더된 DOM=중립 골격, 화면 소스=의미 권위.

권한·컨텍스트 보존

헤드리스 신규 세션이 아니라 사용자가 로그인·메뉴로 진입한 실브라우저에 CDP로 attach. 직타 187위젯(권한버튼 누락) → 메뉴진입 216위젯(전부 렌더).

안정성 — 단일탭 raw 캡처

대형 어드민에서 다중탭 동시 제어 시 Chrome 메모리 크래시 → raw WebSocket CDP로 탭 1개씩 캡처해 회피(실측 대응).

이중 용도 — 캡처 + 협업 콘솔

같은 9222 CDP 다리가 (i)UIS 캡처 (ii)SpecLens 인터랙티브(뷰어 버튼→세션 큐·SR 작업보드·스펙 재생성)를 함께 구동. 페이지 read/write만, 실변경은 사람 게이트.

10핵심 설계원칙 = 실효성의 근거

토큰 최소화

사실(컬럼·타입·링크·조인·필터)은 스크립트가 zero-LLM 생성, 의미(코드값·비즈규칙·필터의미)만 LLM. SCH는 사실 ~70~80%를 토큰 0으로(추정).

컨텍스트 격리

대량 생성은 독립 서브프로세스 디스패처 → 메인 컨텍스트 오버플로 없음 + 실패 자동 재시도.

JIT 그라운딩 (2층)

요약 주입 대신 그래프+소스앵커+구조화 쿼리패턴. 한 마크다운이 JIT 원천이 되지 않게 사람 레이어/기계 레이어 분리.

멱등 · 범용 · 문서동기화

재실행 안전, 스택 중립(DB MCP도 ora/db2/mdb 환경별), 변경 시 정본 문서 강제 동기화로 stale 방지.

--JIT 그라운딩 vs RAG — 무엇이 다른가

축RAG (임베딩 검색)Speclinker JIT (구조그래프+소스앵커)
검색의미 유사도(근사·top-k)결정론적 그래프 순회 + 경로조인
반환청크 텍스트(요약/원문)구조화 사실 + 실코드 위치(file:line)
현행성인덱싱 시점 고정(stale)변경시점 실코드 판독 → 드리프트 0

13정직한 한계

  • 의도/why — 코드가 "무엇을" 하는지는 잡지만 "왜"는 못 잡음(주석 한계)
  • 런타임 암묵 의존 — 동적 SQL·이벤트 리스너·리플렉션은 정적분석 사각지대
  • freshness — 코드가 움직이면 스펙이 낡음 → drift 감지 + JIT 실소스로 완화하나 완전 해결은 아님

→ 이 한계를 인정하고, JIT 실소스 read로 lossy·stale을 구조적으로 보완하는 것이 설계의 핵심.

--현 단계 = PoC · 납품 전 검증 게이트

현 산출물은 개념증명(PoC)이지 납품이 아니다. PoC=작동 가능성 입증, 납품=통제된 효과·비용·우위 입증. 지금 입증된 건 전자 — 효과는 아래 게이트로 검증한다(반증가능).

게이트검증 질문 (반증가능)현재
G1 효과그라운딩이 AIDD 결과를 실제 개선하나 (A/B 통과율·결함 수)하네스만
G2 비용RECON·변경당 토큰·시간이 수용 가능한가미계측
G3 우위JIT가 RAG·소스 직판독 대비 우위인가정성논증
G4~G6의미 충실도 · SOP 가독성 · 규모·범용 경제성미수행

논지: G1이 핵심 — 결과 개선을 못 하면 정교함·산출물 양은 무의미. 우선순위 = 측정(G1→G2→G3) > 기능 확장. 납품 전이는 "산출물 양"이 아니라 게이트 통과로 정의.
