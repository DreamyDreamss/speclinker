01문제 — SM 변경개발의 현실

현행을 모른다

레거시 시스템은 소스 한 파일만 봐선 "지금 어떻게 도는지"를 알 수 없다. 비즈니스 규칙이 컨트롤러→서비스→DAO→SQL→배치에 흩어져 있다.

변경 영향이 안 보인다

공통 테이블 하나 바꾸면 어디가 깨지는지 추적 불가. 보이지 않는 곳에서 회귀가 터진다.

문서가 없거나 낡았다

설계서는 부재하거나 현행과 어긋남(stale). 신규 인력 온보딩·고객 보고에 매번 비용.

AI를 써도 현행을 모른다

AIDD를 도입해도 AI가 운영 시스템의 현행 맥락을 모르면 위험한 변경을 만든다.

04활용 오픈소스·기술과 "왜"

BMAD-METHOD

AIDD를 story-driven + QA 게이트로. FUNC=story, 전용 qa-agent가 독립 컨텍스트로 3-Layer(스펙·보안·회귀) 검증. → 자동 폭주 없는 개발 규율.

MCP (Atlassian·DB)

Jira에서 SR 요구사항·첨부 수집, Confluence 게시, DB information_schema 권위 조회. → 실제 업무 흐름과 직결.

Docsify → SpecLens

역추출 산출물을 무빌드 웹 뷰어로. 대시보드·INF/SCH/UIS·IA 트리·크로스링크 네비게이션. → 사람용 산출물 열람.

tree-sitter · Playwright · Claude

tree-sitter=제로-LLM 정적 소스 스캔, Playwright=화면 캡처(UIS), Claude(Sonnet/Opus)=의미 합성. → 사실은 무료로, 의미만 LLM.

보조: python-pptx/docx/openpyxl(첨부 추출) · pymysql/psycopg2(DB 권위) — 전부 선택 의존성(없으면 우아하게 폴백).

05RECON — 현행을 스펙으로

  • 제로-LLM 정적 스캔 — tree-sitter로 라우터·서비스·DAO·SQL 구조를 토큰 0으로 추출
  • 호출체인 합성 — 컨트롤러→서비스→DAO→쿼리를 따라가 흩어진 비즈니스 규칙을 한 곳에 모음 (소스 한 파일론 안 보이는 것)
  • 디스패처 병렬화 — 독립 서브프로세스로 대량 생성, 메인 컨텍스트 오염 없음, 실패 자동 재시도·멱등
  • 다중 스택 중립 — Java/Spring, Next.js, Python 등 경로 기반 도메인 분류(특정 스택 비종속)

실측(運영 admin 1종): API 657개·테이블 270개 스펙을 자동 역생성.

06산출물 — 이중 용도(사람 + AI)

INF (API 명세)

엔드포인트·요청/응답·비즈니스 규칙·트랜잭션 순서·사이드이펙트 + 근거소스 file:line 앵커

SCH (DB 스키마)

컬럼·타입·인덱스·FK·ERD·코드값·비즈주의 (테이블당 개별 파일)

UIS (화면 설계)

화면 캡처·위젯·API 연결 (Playwright goto 캡처)

FUNC (추적 축)

INF/SCH/UIS를 묶는 기능 단위 — RTM·AIDD의 단일 진실

핵심: 같은 산출물이 사람에겐 읽는 문서, AI에겐 frontmatter·링크·소스앵커로 기계 탐색되는 인덱스.

09핵심 설계원칙 = 실효성의 근거

토큰 최소화

사실(컬럼·타입·링크)은 스크립트가 zero-LLM 생성, 의미(코드값·비즈규칙)만 LLM. SCH는 사실 ~70%를 토큰 0으로(추정).

컨텍스트 격리

대량 생성은 독립 서브프로세스 디스패처 → 메인 컨텍스트 오버플로 없음 + 실패 자동 재시도.

JIT 그라운딩

요약 주입 대신 그래프+소스앵커 → 무손실·최신·정밀. RECON은 "인덱스", 실소스는 "정밀 회귀".

멱등 · 범용 · 문서동기화

재실행 안전, 스택 중립, 변경 시 정본 문서 강제 동기화로 stale 방지.

12정직한 한계

  • 의도/why — 코드가 "무엇을" 하는지는 잡지만 "왜"는 못 잡음(주석 한계)
  • 런타임 암묵 의존 — 동적 SQL·이벤트 리스너·리플렉션은 정적분석 사각지대
  • freshness — 코드가 움직이면 스펙이 낡음 → drift 감지 + JIT 실소스로 완화하나 완전 해결은 아님

→ 이 한계를 인정하고, JIT 실소스 read로 lossy·stale을 구조적으로 보완하는 것이 설계의 핵심.
