# Speclinker 개선 마스터 플랜 — 논문 한계 → 빌드 가능 개선

> 출처: `docs/report/speclinker-paper.md` Part A(결함 10) + §6 타당성 위협.
> 본 문서는 4개 트랙으로 분해한 **마스터 플랜**이다. 각 트랙은 독립 spec→plan→실행이 가능하며, 아래는 트랙별 컴포넌트·범위·검증·의존성을 task 수준으로 정의한다.

**목표:** 논문이 지적한 한계(측정 부재 / JIT 입구·출구 약함 / 현행성 / 산출물 품질)를 실제 코드로 메워, "이중 역할 산출물로 AIDD 실현" 명제를 *논증*에서 *검증 가능*으로 끌어올린다.

**불변식:** RECON 산출물 형식·경로 무변경. 기존 파이프라인(INF/SCH/UIS/FUNC, AIDD, sl-change) 무파괴. 신규는 전부 선택/증분. 범용성(스택중립)·멱등·doc-sync 준수.

---

## 트랙 우선순위·의존성

```
T1 JIT 입구·출구 ──┐ (P0, 제품 실효 직결)
                   ├─▶ T3 측정 하네스가 T1 효과를 정량화 (H3 ripple, H4 AIDD)
T2 현행성 게이트 ──┘ (P1, 안전성)
T3 측정 하네스      (P1, 논문 입증 — H1 충실도는 독립, H4는 T1 후 의미)
T4 산출물 품질      (P2, T1/T3와 병행 가능)
```
- **권장 순서**: T1 → T3(충실도 H1 먼저) → T2 → T4. T1은 T3의 H3/H4 측정 대상이 되므로 선행.
- 각 트랙은 자체 `docs/superpowers/specs|plans/...` 로 분리 실행. 본 문서는 상위 색인.

---

# T1. JIT 입구·출구 강화 (P0)

**대응 결함:** #3(엔티티 자동추출), #4(ripple 랭킹·전이). **명제 기여:** JIT 그라운딩의 재현율(입구)·정밀도(출구).

## T1-A. 엔티티 자동추출 (`scripts/extract_entities.py`, 신규)
- **입력:** SR 본문(`00_요구사항.md`) + 첨부 추출(`_extracted.md`) + (선택)그래프(테이블/INF/path 어휘).
- **처리(zero-LLM 1차 + LLM 보강 2차):**
  1. 그래프 어휘 사전 매칭 — 텍스트에서 알려진 테이블명/INF-ID/엔드포인트 path 직접 검출(대소문자·스네이크 정규화).
  2. 휴리스틱 — 대문자_스네이크(테이블 후보), `/path/seg`(엔드포인트), 한글 화면명↔UIS 매핑.
  3. (NETWORK/토큰 허용 시) LLM 보강 — 도메인 용어→엔티티 정규화(예: "환불"→refund 관련 테이블군). 옵션.
- **출력:** `docs/변경관리/{SR-ID}/_entities.json` = `{tables:[], infs:[], paths:[], screens:[], terms:[]}` + 신뢰도/출처.
- **검증:** 합성 SR(테이블 2·path 1 포함) → 정확 검출. 누락률(재현율) 로그.

## T1-B. ripple 랭킹·전이 (`build_change_context.py` 확장)
- **관련도 점수:** 각 영향 노드에 score = f(시드 거리 1/2홉, 시드 직접매칭=1.0, 공유테이블 편재도 역가중).
- **편재 자원 감쇠(ubiquity damping):** 테이블 t의 `len(table_to_inf[t])`가 클수록(공통코드 등) ripple 기여 가중 ↓. 임계 초과 테이블은 "광역 공통자원" 별도 섹션으로 격리(176개 나열 방지).
- **전이 확장(옵션 `--hops N`):** INF→테이블→그 테이블을 *쓰는*(write) INF까지 2홉. 기본 1홉.
- **출력 변경:** `_asis_brief.md`에 **상위 K개 위험(점수순)** + "광역 공통자원(JT_CODE 등 N건, 영향 광범위 — 개별 검토 권장)" 분리.
- **검증:** 합성 — 공통테이블(다수 사용) + 전용테이블(소수) 혼합 → 전용테이블 사용처가 상위, 공통은 격리 섹션. 기존 test_change_context 회귀 유지.

## T1 산출/순서: 자체 spec→plan. T1-A 먼저(출력이 B의 입력), 그 다음 T1-B. sl-change Step 5에 extract_entities → build_change_context 연결.

# T2. 현행성 게이트 (P1)

**대응 결함:** #5(freshness 비강제). **명제 기여:** 그래프/스펙 stale로 인한 영향슬라이스 오류 방지.

## T2-A. drift 게이트 (`sl-change` Step 0.5 신설)
- 변경 착수 전, **영향 도메인의 스펙-소스 drift**를 검사(기존 `sl-drift` 로직 재사용). stale이면 경고 + 사용자 확인(차단 옵션).
- 검사 범위: T1-A 엔티티의 도메인. 전체 drift 아님(빠르게).

## T2-B. 영향영역 그래프 재빌드 (`spec_graph_build` 증분)
- drift 감지 도메인만 **RECON 부분 재실행 또는 frontmatter 재스캔**으로 그래프 갱신.
- 코드에 신규 테이블/INF가 생겼으나 스펙 미반영 시 "그래프 누락 가능" 경고 + 해당 소스 직접 스캔 폴백.
- **검증:** 스펙에 없는 테이블을 소스에서만 추가한 픽스처 → 게이트가 "그래프 누락" 경고.

## T2 산출: 자체 spec→plan. T1 이후(엔티티로 검사범위 한정).

---

# T3. 측정 하네스 (P1) — 논문 H1·H3·H4 입증

**대응 결함:** #1(충실도), #2(AIDD 결과). **명제 기여:** 효과성 *논증*→*검증*.

## T3-A. 충실도 하네스 (`scripts/eval_fidelity.py`, 신규) — H1
- **샘플링:** INF/SCH N개 무작위 추출 → 전문가 주석용 워크시트 생성(스펙 추출 항목 vs 빈칸 정답).
- **비교:** 주석 완료본 입력 → 추출 비즈규칙·코드값·테이블 사용처의 precision/recall/F1 계산·리포트.
- **자동 보조:** 테이블 사용처는 소스 grep 정답(SQL의 테이블 토큰)과 자동 대조(부분 자동화).
- **출력:** `docs/report/eval/fidelity_{date}.md` (지표 + 오류 사례).
- **주의:** 정답은 사람 주석 필요(완전 자동 아님). 하네스가 *측정 가능*하게 만든다.

## T3-B. AIDD 결과 하네스 (`scripts/eval_aidd.py`, 신규) — H4
- **설계:** 동일 FUNC 집합을 (a)풀 그라운딩(JIT) vs (b)소스만 구현 → 각 산출 코드의 **테스트 통과율 + 리뷰 결함 수** 비교.
- **독립 오라클:** qa-agent(스펙 오라클) 대신 **실제 테스트 실행 + 인간/별도 LLM 리뷰**를 정답으로(순환 회피).
- **출력:** `docs/report/eval/aidd_{date}.md` (A/B 표).
- **주의:** 실행·리뷰 루프 필요(반자동). 픽스처 FUNC 소수로 파일럿.

## T3 산출: 자체 spec→plan. H1(T3-A)은 독립적으로 즉시 가능, H4(T3-B)는 T1 후 의미.

# T4. 산출물 품질 (P2)

**대응 결함:** #7(lossy² 코드값), #6(SOP 가독성). **명제 기여:** 이중 역할의 인간 SOP 측면 강화.

## T4-A. 코드값 사실 역인용 (`ddd-db-agent` enrichment 강화)
- 코드값 의미를 *요약의 요약*이 아니라, **공통코드 테이블(JT_CODE 등) 조회 + 소스 if/switch 분기에서 사실로 역인용**하고 **출처(file:line) 표기 강제**.
- 추정/불명은 `[미확인]` 명시(허위 의미 부여 금지).
- **검증:** 공통코드 참조 컬럼 → 값·의미가 공통코드 테이블 출처와 함께 채워짐.

## T4-B. SOP 내러티브 레이어 (`scripts/build_domain_overview.py`, 신규)
- 도메인별 **개념·업무흐름 개요**(서술적 how-to 지향)를 기계 인덱스와 **분리**해 생성: 도메인 목적, 핵심 엔티티 관계, 대표 시나리오(주요 INF 체인), 신규자 진입점.
- 기존 frontmatter/앵커(기계용)는 불변 — 인간용 레이어만 추가(이중 레이어 명시적 분리, 결함 #10 대응).
- **검증:** 도메인 개요에 "이 도메인은 무엇을 하는가 + 대표 흐름"이 포함. 신규자 이해도 실험(H2)은 T3 연계.

## T4 산출: 자체 spec→plan. T1/T3과 병행 가능.

## doc-sync (각 트랙 실행 시)
- `scripts/README.md`(신규 스크립트), `CLAUDE.md`(버전노트), `docs/RECON_PIPELINE.md`(T2-B/T4 RECON 영향), `skills/sl-change/SKILL.md`(T1-A/T1-B/T2-A), `docs/report/speclinker-paper.md`(측정 결과 §5.1로 승격).

## 실행 권고
1. **T1-A → T1-B** (P0, 즉시 실효 + T3 측정 대상)

3. **T2-A → T2-B** (안전성)
4. **T3-B, T4-A, T4-B** (병행)

각 트랙은 본 마스터 플랜을 상위로 두고 자체 spec→plan→실행으로 진행한다.
